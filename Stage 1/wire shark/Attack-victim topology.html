Your Complete Wireshark Guide — Teacher Mode 🎓

Module 1 — What Wireshark really is (and why it matters in cybersecurity)

Think of it this way: Every time your computer talks to another device — sending a login request, loading a website, pinging a server — it breaks that communication into tiny chunks called packets. Wireshark is a tool that grabs every single one of those packets off the wire and lets you read them like an open book.

In cybersecurity, this is your superpower. You can see exactly what is happening on a network, who is talking to whom, what data is being sent, and whether something malicious is going on.

What Wireshark is used for in cybersecurity:

  • Detecting intrusions and suspicious traffic patterns
  • Capturing and analyzing malware communication with command-and-control servers
  • Spotting credential theft over unencrypted protocols
  • Investigating security incidents after the fact (digital forensics)
  • Verifying that your firewall rules are actually working
  • Catching ARP poisoning, port scans, and DoS attacks in real time

Understanding the Wireshark interface — 3 panes:

The first pane at the top is the Packet List — every captured packet appears here as one row showing time, source IP, destination IP, protocol, and a brief info summary.

The middle pane is the Packet Details — when you click any packet, this expands all its protocol layers. You'll see the Ethernet frame, the IP header, the TCP/UDP header, and the actual application data, all stacked like the OSI model layers.

The bottom pane is the Packet Bytes — the raw hexadecimal and ASCII representation of the packet. This is what the data actually looks like in memory.

Two types of filters you must know from day one:

A capture filter is applied before capturing — it tells Wireshark to only grab certain packets. Example: host 192.168.1.1 will only capture traffic to/from that IP. This saves resources.

A display filter is applied after capturing — it hides packets you don't want to see right now but keeps them in memory. Example: http shows only HTTP packets. You can change this anytime without losing data.

🧪 Lab 1 — Your First Packet Capture

Objective: Capture your own browser traffic and read it packet by packet.

Step 1: Open Wireshark. Select your active network interface (it will show packets moving — you'll see the activity graph flickering).

Step 2: Hit the blue shark fin button to start capturing. Now open your browser and go to http://neverssl.com (this is a plain HTTP site — no encryption, so you can read everything).

Step 3: Stop the capture after 10 seconds. In the display filter bar type http and press Enter. You'll now see only HTTP packets.

Step 4: Click on a GET request packet. In the middle pane, expand Hypertext Transfer Protocol. You'll see the exact HTTP request your browser sent — including the Host header, User-Agent (your browser version), and the URL requested.

Step 5: Right-click that packet → Follow → TCP Stream. A window will open showing the entire conversation between your browser and the server in plain text. This is exactly how an attacker would read your traffic on an unencrypted network.

What to observe: Notice how you can see the browser type, the URL, and even the server's response. This is why HTTP (not HTTPS) is dangerous on public Wi-Fi.

Windows Terminal equivalent — doing the same with netstat:

Open PowerShell as administrator and run:

netstat -an

This shows all active connections on your machine. To watch only established TCP connections in real time:

netstat -an 5

This refreshes every 5 seconds. While your browser is loading that page, you'll see the connection appear with the remote IP address. To find the hostname from that IP:

nslookup <ip-address>

Module 2 — Protocol Analysis Deep Dive

Now you understand what you're looking at. Let's go deeper into the protocols you'll encounter every day as a network security engineer.

TCP Three-Way Handshake — the foundation of everything:

Every TCP connection starts with three packets. SYN (client says "I want to connect"), SYN-ACK (server says "OK, I'm ready"), ACK (client says "great, let's go"). In Wireshark, filter with tcp.flags.syn == 1 to see only these handshake packets. If you see a flood of SYN packets with no SYN-ACK replies, that's a SYN flood DoS attack.

DNS — the phonebook of the internet:

When you type google.com, your computer first asks a DNS server "what's the IP for google.com?" Filter with dns in Wireshark. You'll see DNS Query packets (type A for IPv4, AAAA for IPv6) and DNS Response packets. In cybersecurity, malware often uses DNS to find its command-and-control server. Unusual DNS requests to random-looking domains are a red flag.

HTTP vs HTTPS:

HTTP traffic is completely readable in Wireshark — you can see the URL, headers, cookies, and even POST data (including passwords). Filter http to see it. HTTPS traffic is encrypted — you'll only see TLS handshake packets, not the content. Filter tls to see TLS traffic. The fact that you can see HTTPS but not read it is actually a security success story.

ARP — where trouble often starts:

ARP maps IP addresses to MAC addresses on your local network. Filter arp in Wireshark. Normal ARP looks like "Who has 192.168.1.1? Tell 192.168.1.50." Dangerous ARP looks like the same machine constantly broadcasting fake mappings — that's ARP poisoning, and it's how MITM attacks work.

Module 3 — Filtering Mastery

Filtering is what separates a beginner from a professional. When you're dealing with thousands of packets from a real network, your ability to isolate exactly what matters is everything.

Capture filters (Berkeley Packet Filter syntax):

These go in the capture filter bar before you start. They use BPF syntax:

host 192.168.1.1 — only traffic to/from this IP

net 192.168.1.0/24 — only traffic within this subnet

port 80 — only HTTP traffic

port 443 — only HTTPS traffic

not port 22 — everything except SSH

tcp and host 10.0.0.5 — only TCP traffic from that specific host

Display filters (Wireshark's own syntax — more powerful):

ip.addr == 192.168.1.1 — traffic involving this IP

ip.src == 192.168.1.1 — only traffic FROM this IP

tcp.port == 443 — HTTPS traffic

http.request.method == "POST" — only HTTP POST requests (often contains login data)

dns.qry.name contains "google" — DNS queries containing "google"

tcp.flags.reset == 1 — all TCP reset packets (connection interruptions — could indicate a scan or firewall block)

frame.len > 1000 — large packets (useful to find file transfers)

Combining filters with logic:

&& means AND — ip.src == 10.0.0.1 && tcp.port == 80

|| means OR — tcp.port == 80 || tcp.port == 443

! means NOT — !arp (hide all ARP traffic)

Module 4 — Cybersecurity Threat Detection (The Good Stuff)

This is where everything clicks together. You now know protocols, you know filters, and you're working at a power grid company where these attacks are real threats.

Detecting a port scan:

When an attacker scans your network with a tool like Nmap, they send SYN packets to many ports rapidly. In Wireshark, this looks like one source IP sending SYN packets to the same destination IP but on hundreds of different ports within seconds. Filter: tcp.flags.syn == 1 && tcp.flags.ack == 0 then look for one source IP appearing repeatedly in the packet list. Also look for a large number of tcp.flags.reset == 1 responses — these are closed port responses.

Detecting ARP Poisoning (MITM attack):

Filter arp. Look for one MAC address claiming to be many different IP addresses, or the same IP address appearing in multiple ARP replies from different MACs. The attack works by an attacker flooding the network with fake ARP replies telling everyone "I am the router" — this redirects all traffic through the attacker's machine. In Wireshark, you'll see duplicate ARP replies for the same IP from different MACs.

Detecting credential theft over plain text protocols:

Filter ftp — you can see FTP usernames and passwords in plain text in the Info column. Filter telnet — entire terminal sessions are readable. Filter http.request.method == "POST" — look at the packet details and expand the HTML form data. You may see username=admin&password=1234 in the clear.

Detecting DoS/DDoS:

Go to Statistics → I/O Graph. A legitimate network shows a relatively steady line. A DoS attack shows a sudden massive spike in packets per second from one or a few sources. Filter icmp and look for ICMP flood — thousands of ping packets from one IP. Filter udp and look for UDP flood attacks on specific ports.

Malware Command-and-Control (C2) traffic patterns:

Malware usually connects to a remote server for instructions. Signs to look for: regular periodic connections to an unusual IP (beaconing behavior — every 30 seconds or 60 seconds exactly), DNS requests for randomly-generated domain names (domain generation algorithms — they look like xj3kp9mq.com), large outbound data transfers to unknown IPs (data exfiltration), and HTTP POST requests to strange domains with encoded payloads in the body.

Module 5 — Intermediate: Forensics, tshark, and Automation

At this level, you stop just looking at traffic and start building workflows that make you faster and more effective as a security engineer.

PCAP forensics — extracting files from traffic:

When someone downloads a file over HTTP (unencrypted), every byte of that file passes through the network as packets. Wireshark can reassemble and extract it. Go to File → Export Objects → HTTP. A list of all files transferred over HTTP will appear. You can save any of them — images, executables, documents — directly from the packet capture. This is how forensic analysts recover files transferred during an incident.

tshark — Wireshark's command-line brain:

tshark is the terminal version of Wireshark. This is where you become truly powerful, because you can automate analysis, run it on servers, and script everything.

Read a PCAP file and display all packets:

tshark -r capture.pcap

Apply a display filter from command line:

tshark -r capture.pcap -Y "http.request.method == POST"

Extract only specific fields (perfect for scripting):

tshark -r capture.pcap -Y "dns" -T fields -e dns.qry.name

This outputs only the DNS query names — one per line. You can pipe this to PowerShell to count or filter further.

Statistics and I/O graphs for incident reports:

Go to Statistics → Protocol Hierarchy. This gives you a breakdown of all protocols in your capture by percentage. In a healthy network, you'll see mostly TCP with HTTP/HTTPS. If you suddenly see a large percentage of ICMP or unknown UDP, that's a finding worth reporting.

Go to Statistics → Endpoints. This lists every IP that appeared in the capture with byte counts. Sort by bytes sent — the top talker in an unexpected direction is a red flag.

Writing an incident report from Wireshark findings:

Every finding you make in Wireshark needs to be documented properly. Structure it as: what you observed (the raw data — filter, timestamp, source/destination IP), what it means (the interpretation — this matches pattern X), and what you recommend (block IP, isolate host, escalate to SOC).

Module 2 — Protocol Analysis Deep Dive

Why protocols matter in security

Think of protocols as languages. When two computers talk, they agree on a language first — TCP, DNS, HTTP, ARP. As a security engineer, your job is to understand every language so fluently that when something abnormal gets said, you catch it instantly. Right now at Power Grid, real conversations in these protocols are happening on your network every second. Let's learn to read them.

TCP — The Reliable Workhorse

TCP is connection-oriented, meaning before any data moves, both sides shake hands and agree to talk. This is the 3-way handshake. Use the TCP Handshake tab in the widget above to study the visual — it's important you can draw this from memory.

In Wireshark, every TCP packet has a Flags field inside the TCP layer. The important flags are SYN (synchronise — open connection), ACK (acknowledge — I received your last packet), FIN (finish — close connection gracefully), RST (reset — close connection immediately, often forcefully), and PSH (push — send data now without buffering).

The Sequence Number and Acknowledgement Number fields are how TCP tracks every single byte. If packet 5 doesn't arrive, TCP knows because the ACK number doesn't advance. It requests retransmission. In Wireshark, you'll sometimes see packets labeled [TCP Retransmission] in red — these mean a packet was lost and had to be resent. A healthy network has very few of these. A congested or attacked network has many.

Apply tcp.analysis.flags in the display filter — this highlights every TCP anomaly Wireshark detected. On a suspicious capture, this filter immediately shows you where the problems are.

DNS — Where Security Begins and Ends

DNS is the very first thing that happens before any internet connection. Before your browser sends a single byte to a website, it asks a DNS server "what's the IP for this domain?" That query goes to your configured DNS server (usually your router or your ISP's server) over UDP port 53.

The security implications are enormous. Malware uses DNS to find its command-and-control server. If you block or monitor DNS, you can detect and stop malware before it ever makes its first connection. This is why enterprise security tools like DNS firewalls exist — they are essentially Wireshark-style DNS monitoring at scale.

What malicious DNS looks like: legitimate domains are short and readable (google.com, microsoft.com). Malware-generated domains look like xk9p2mq3.xyz or a7b2c9d1e4f.com — these are created by Domain Generation Algorithms (DGAs). If you see a machine making DNS queries for random-looking domains, that machine is likely infected.

Another attack: DNS tunnelling. Malware encodes stolen data inside DNS query names and sends it out to attacker-controlled DNS servers. The query looks like c3RvbGVuZGF0YQ.evil.com — that long random string is base64-encoded stolen data. Wireshark's DNS dissection will show you the full query name — abnormally long DNS names are a red flag.

HTTP vs HTTPS — Know the Difference in the Packet

When you filter http in Wireshark, you see the full application layer conversation — GET requests, POST requests with form data, server responses with status codes (200 OK, 404 Not Found, 302 Redirect), cookies in the headers, and even the HTML body of the page.

When you filter tls, you see encrypted traffic. You can still read the TLS handshake — it shows which TLS version is used, the cipher suite negotiated, and importantly the SNI (Server Name Indication) field which reveals which website is being visited even through encryption. The body is encrypted but the destination is not hidden. This matters for security monitoring — you can know a machine visited malware-c2.ru even without decrypting the traffic.

ARP — The Local Network's Weak Spot

ARP is brutally simple and brutally insecure. When your machine wants to talk to 192.168.1.1 (your router), it broadcasts "Who has 192.168.1.1? Tell 192.168.1.50". The router responds "192.168.1.1 is at AA:BB:CC:DD:EE:FF". Your machine caches this and uses that MAC address for all future packets to the router.

Here's the vulnerability — ARP has no authentication. Any machine on the network can send an unsolicited ARP Reply saying "192.168.1.1 is at MY-MAC-ADDRESS". If your machine accepts this (and most do by default), it will now send all your internet traffic to the attacker instead of the real router. The attacker forwards it on so you don't notice — but reads every packet in transit. This is a Man-in-the-Middle attack, and it's been done on power and industrial networks.

The Wireshark detection signature: filter arp and look for duplicate IP-to-MAC mappings. Wireshark itself will flag this — you'll see a packet with the warning [Duplicate IP address detected] in the Info column.

ICMP — The Recon Tool Attackers Love

ICMP is the protocol behind ping and tracert. It has no ports — it works at Layer 3. Type 8 = Echo Request (ping sent). Type 0 = Echo Reply (ping received). Type 3 = Destination Unreachable. Type 11 = Time Exceeded (what tracert uses).

Attackers use ICMP for network mapping — ping sweep means sending ICMP Echo Requests to every IP in a range to find which ones are alive. In Wireshark, this looks like one source IP sending ICMP Type 8 to sequential destination IPs like 192.168.1.1, 192.168.1.2, 192.168.1.3 in rapid succession. Filter icmp and sort by Source — if one IP is generating dozens of ICMP requests per second to different destinations, that's reconnaissance.

FTP and Telnet — The Dangerous Legacies

Here's why these matter specifically at Power Grid Corporation: industrial control systems, SCADA devices, older PLCs, and legacy infrastructure very commonly still use FTP and Telnet. These protocols send everything in plain text — including usernames and passwords.

Filter ftp in Wireshark on an industrial network capture. You'll see USER admin and PASS password123 sitting in the Info column in plain text. Filter telnet and you can read the entire terminal session someone had with a device — every command they typed, every output they received.

If you ever find FTP or Telnet in use on your Power Grid network, that is a critical security finding. Report it. Those devices need SSH and SFTP instead.

How it all connects — the full picture of one web request

When you visit neverssl.com, here is the complete sequence of protocol events you should now be able to trace in Wireshark:

First, an ARP request goes out to find the MAC address of your default gateway (your router). Then a DNS query goes out over UDP port 53 asking for neverssl.com's IP. The DNS response comes back with the IP. Then a TCP SYN packet goes to that IP on port 80. The server sends SYN-ACK. You send ACK. Connection established. An HTTP GET request goes out over that TCP connection. The server responds with HTTP 200 OK and the HTML. A TCP FIN closes the connection.

Every single one of those steps is visible in Wireshark. That is what makes it the most powerful tool in a network security engineer's hands.

Module 3 — Filtering Mastery

Why filtering is THE skill

Imagine you're at Power Grid and something suspicious happens on the network. Wireshark captures 500,000 packets in 60 seconds. Without filtering, you're looking for a needle in a haystack the size of a football field. With the right filter, you find it in 3 seconds. That is the difference between a junior analyst and a senior one — not knowledge of protocols, but speed of isolation.

The Two Types of Filters — Understand This Deeply

Capture Filters (BPF — Berkeley Packet Filter syntax)

Set BEFORE you start capturing. Wireshark throws away any packet that doesn't match. This saves memory and disk space on long captures. The syntax is different from display filters — it's older, simpler.

host 192.168.1.5          → only traffic to/from this IP
net 10.10.0.0/16          → only traffic within this subnet
port 80                   → only port 80 traffic
tcp and port 443          → only HTTPS TCP traffic
not port 22               → everything EXCEPT SSH
host 10.0.0.1 and port 80 → HTTP from one specific machine

Critical warning: If you set a capture filter too narrow, you permanently miss packets. You can never get them back. Use capture filters only when you know exactly what you want. When in doubt, capture everything and use display filters.

Display Filters (Wireshark's own syntax)

Applied AFTER capture. Packets are only HIDDEN, never deleted. You can change, clear, and reapply these as many times as you want without losing data. This is your real investigation tool.

ip.addr == 192.168.1.5             → traffic involving this IP
ip.src == 192.168.1.5              → only FROM this IP
ip.dst == 192.168.1.5              → only TO this IP
tcp.port == 443                    → HTTPS on either side
tcp.dstport == 80                  → destination port 80 only
http                               → all HTTP traffic
http.request.method == "POST"      → login forms, data submissions
http.request.method == "GET"       → page requests
dns                                → all DNS traffic
dns.qry.name contains "evil"       → DNS queries with a substring
tls                                → all encrypted HTTPS traffic
arp                                → all ARP traffic
icmp                               → all ping/traceroute traffic
ftp                                → FTP command channel
frame.len > 1000                   → large packets (transfers)
tcp.flags.syn == 1                 → all SYN packets
tcp.flags.reset == 1               → all TCP RST (refused/broken connections)
tcp.analysis.flags                 → ALL TCP anomalies Wireshark detected automatically

Logical Operators — Combining Filters

This is where you get surgical. Three operators:

&& means AND — both conditions must be true || means OR — either condition is enough ! means NOT — exclude this

Real examples:

ip.src == 10.0.0.5 && http
→ HTTP traffic FROM a specific machine only

tcp.port == 80 || tcp.port == 443
→ All web traffic (HTTP + HTTPS)

!arp && !dns
→ Hide all ARP and DNS noise — see only application traffic

ip.src == 10.0.0.5 && tcp.flags.syn == 1 && tcp.flags.ack == 0
→ Port scan detection filter — SYN packets from one source

http.request.method == "POST" && http.content_type contains "form"
→ Form submissions over HTTP — potential credential exposure

dns.qry.name && frame.len > 100
→ Abnormally large DNS queries — DNS tunnelling indicator

!broadcast && !multicast && !arp
→ The "clean traffic" filter — removes all network housekeeping noise

The contains and matches operators

These are for string and pattern searching inside packet fields.

dns.qry.name contains "microsoft"
→ Any DNS query that includes "microsoft" in the domain name

http.host contains "login"
→ HTTP requests going to hosts with "login" in the hostname

dns.qry.name matches "[a-z]{15,}"
→ Regex match — DNS queries with 15+ consecutive lowercase letters
→ This is a DGA (Domain Generation Algorithm) malware detector

The matches operator accepts full regular expressions. This is advanced but powerful for threat hunting.

Security-Specific Power Filters — Use These at Power Grid

Port scan detection:

tcp.flags.syn==1 && tcp.flags.ack==0

Apply this. Look at the Source IP column. If one IP appears dozens of times sending to different destination ports — that's a port scan. If it's sending to the same destination port many times per second — that's a SYN flood DoS.

ARP poisoning detector:

arp.duplicate-address-detected

Wireshark flags this automatically. This filter shows you every packet where Wireshark detected that the same IP was claimed by two different MAC addresses. One such packet is suspicious. Ten is an active attack.

Data exfiltration hunt:

frame.len > 1400 && ip.dst != [your gateway IP]

Large packets going to an external IP that isn't your known gateway. Could be file uploads or data theft.

Credential sniffing hunt:

ftp || telnet || http.request.method == "POST"

Three in one — FTP plain text passwords, Telnet sessions, and HTTP form submissions. On any industrial network, FTP and Telnet findings are critical security issues.

Statistics — Your Second Set of Eyes

Beyond filters, Wireshark has a Statistics menu that gives you aggregated views without reading individual packets.

Statistics → Protocol Hierarchy: Shows every protocol in the capture as a percentage of total traffic. A normal network is mostly TCP/TLS. If you see 40% ICMP, something is wrong. If you see protocols you don't recognise, investigate them first.

Statistics → Conversations: Lists every pair of IPs that communicated, with total bytes transferred. Sort by Bytes (descending). The top talker to an unknown external IP is your immediate focus in an incident. This one view can identify data exfiltration faster than any filter.

Statistics → Endpoints: Every single IP that appeared in the capture with their total traffic volume. An IP you've never seen before with high byte counts = investigate immediately.

Statistics → I/O Graph: Traffic volume over time as a graph. A flat line is a healthy network. A sudden vertical spike is a DoS, a flood, or a burst transfer. The moment of a security incident shows up visually here before you even look at individual packets.

Module 4 — Cybersecurity Threat Detection in Wireshark

The mindset shift

Modules 1–3 taught you to READ the network. Module 4 teaches you to HUNT it. You're no longer a student observing packets — you're a security engineer looking for attackers. Every filter you learned, every protocol you understand, every OSI layer you can navigate — it all becomes your weapon right now.

At Power Grid Corporation, the threats are real. Industrial networks get targeted. SCADA systems get probed. This module is directly applicable to your internship.

Attack 1 — Port Scanning (Reconnaissance)

Before any attack, an attacker maps your network. They need to know what's alive and what ports are open. This is called reconnaissance, and it leaves a very clear signature in Wireshark.

What it looks like: One source IP sends SYN packets to the same destination IP but across dozens or hundreds of different port numbers — all within a few seconds. The destination responds with either SYN-ACK (port is open — attacker notes this) or RST (port is closed).

Wireshark filter to detect it:

tcp.flags.syn==1 && tcp.flags.ack==0

Apply this. Sort the packet list by Source IP. If one IP appears many times sending to different destination ports — that's a scan. A legitimate user opening web pages generates maybe 5–10 SYN packets. A port scanner generates hundreds per second.

The RST signature: Pair it with:

tcp.flags.reset==1

A flood of RST packets from the target back to the scanner means many ports are closed — the scanner is getting rejected. This RST flood is the target's side of a port scan.

At Power Grid specifically: Any device on your OT (operational technology) network receiving SYN packets from an IT network IP is a critical alert. IT and OT should have strict segmentation — SYN packets crossing that boundary mean segmentation has failed or is being probed.

Attack 2 — ARP Poisoning / Man-in-the-Middle

You learned ARP in Module 2. Now here's how it becomes an attack weapon.

How the attack works: The attacker sends fake ARP Replies to every device on the subnet saying "The router's IP (192.168.1.1) is at MY MAC address." Every device updates its ARP cache. Now all internet traffic flows through the attacker's machine. The attacker forwards it to the real router so nothing seems broken — but reads every packet in transit.

Wireshark filter:

arp

Then look for: the same MAC address appearing in multiple ARP Replies claiming to be different IP addresses, OR the same IP address appearing in ARP Replies from two different MAC addresses. Wireshark will actually flag this automatically with a yellow warning in the Info column: [Duplicate IP address detected for 192.168.1.1 - also in use by aa:bb:cc:dd:ee:ff]

The definitive filter:

arp.duplicate-address-detected

If this returns any results — someone is either misconfigured OR actively poisoning. Either way, investigate immediately.

At Power Grid: ARP poisoning on an industrial network could allow an attacker to intercept commands sent to PLCs or SCADA systems. A spoofed command to a substation relay is not just a data breach — it's a physical safety event.

Attack 3 — SYN Flood (DoS Attack)

A port scan sends SYNs to many ports. A SYN flood sends thousands of SYNs to the SAME port. The goal is to exhaust the server's connection table — every half-open SYN connection consumes memory until the server can't accept any more legitimate connections.

What it looks like: Same filter as port scan:

tcp.flags.syn==1 && tcp.flags.ack==0

But the difference is in the pattern. Port scan = one source IP, many destination ports. SYN flood = one source IP, one destination port, extremely high packet rate. Go to Statistics → I/O Graph while this filter is active. A SYN flood shows as a vertical spike that stays high — sustained high-rate SYN packets with no corresponding ACK completions.

The confirming signature: The server never sends SYN-ACK back (or sends a few then stops). The connections never complete. Filter:

tcp.flags.syn==1 && tcp.flags.ack==1

If there are very few or zero SYN-ACKs corresponding to the flood of SYNs — the server is overwhelmed and dropping them.

Attack 4 — Credential Theft Over Plain Text Protocols

This is the most directly dangerous thing on legacy infrastructure networks, and Power Grid is full of legacy infrastructure.

FTP credentials in plain text:

ftp

In the Info column you will literally see Request: USER admin and Request: PASS password123. No decryption needed. No special tools needed. The credentials are sitting there readable to anyone on the same network segment.

Telnet session hijacking:

telnet

Follow TCP Stream on any Telnet packet. You will see the entire terminal session — every command typed, every output returned, every file listed. An attacker capturing this can replay the session or use the credentials to log in directly.

HTTP form credential capture:

http.request.method == "POST"

Click any POST packet. Expand Hypertext Transfer Protocol → HTML Form URL Encoded. You may see: username=admin&password=PowerGrid2024. This is the single most common way credentials get stolen on networks that haven't moved to HTTPS.

At Power Grid: this is your highest-priority finding. If you see FTP or Telnet in use, document it. Write the source IP, destination IP, timestamp, and protocol. Escalate it. Those devices need to be migrated to SSH/SFTP immediately.

Attack 5 — Malware Command-and-Control (C2) Traffic

Malware doesn't just sit quietly. After infecting a machine, it calls home to receive instructions. This outbound communication has specific patterns you can detect in Wireshark.

Beaconing behaviour: Malware contacts its C2 server at regular intervals — every 30 seconds, every 60 seconds, every 5 minutes exactly. Go to Statistics → Conversations. Sort by IPv4. If you see one internal IP making repeated connections to one external IP at perfectly regular intervals — that's beaconing. Legitimate apps don't communicate with machine-clock precision.

Filter for periodic connections:

ip.dst == [suspicious external IP]

Then look at the Time column. If packets appear at 30.000, 60.000, 90.000 seconds — the regularity itself is the red flag.

DGA domain detection (Domain Generation Algorithm): Malware generates random-looking domain names to find its C2 server. Filter:

dns

Look at the dns.qry.name column. Legitimate names look like google.com, windows.com, microsoft.com. DGA names look like xk9p2mq3rv.xyz, a7b2c1d9e4f.top, qpzmwoxn.cc. The random strings and unusual TLDs (.xyz, .top, .cc, .pw) are the signature.

DNS tunnelling (data exfiltration over DNS): Malware encodes stolen data inside DNS query names:

dns.qry.name && frame.len > 100

A normal DNS query is 30–60 bytes. A DNS tunnelling packet is 100–500 bytes because the query name itself contains encoded data like c3RvbGVuZGF0YQ.c2Vjb25kYnl0ZXM.attacker.com. The subdomain before the actual domain is the encoded payload.

C2 over HTTP — the hardest to detect:

http.request.method == "POST" && http.content_length > 500

Malware often uses HTTP POST requests to send data back to the C2 server because HTTP blends in with normal traffic. The POST body contains encoded or encrypted data. Look for POSTs to unusual domains, at regular intervals, with encoded-looking payloads in the body.

  • Add 2 PCs and 1 Switch from the bottom panel
  • Connect PC0 to Switch with a copper straight-through cable
  • Connect PC1 to Switch with a copper straight-through cable
  • Click PC0 → Desktop → IP Configuration → set IP 192.168.1.10, Subnet 255.255.255.0
  • Click PC1 → Desktop → IP Configuration → set IP 192.168.1.20, Subnet 255.255.255.0

Exercise A1 — Watch a clean TCP handshake in slow motion

Step 1: Switch to Simulation Mode (clock icon, bottom right). Check that HTTP is in the Event List Filter — click "Show All / None" then select only TCP and HTTP.

Step 2: Click PC0 → Desktop → Web Browser. Type 192.168.1.20 in the address bar and hit Go.

Step 3: Click the Play button (slow speed). Watch the packet envelopes travel across the topology. Click Capture/Forward (the single step button) to move one packet at a time.

Step 4: When a packet envelope appears, click on it. A window opens showing the OSI Model tab and the Inbound/Outbound PDU Details tab.

Click PDU Details. You will see:

  • Packet 1: TCP, SYN flag = 1, ACK flag = 0, Seq = 0 — this is the handshake opening
  • Packet 2: TCP, SYN flag = 1, ACK flag = 1, Seq = 0, Ack = 1 — server confirms
  • Packet 3: TCP, SYN flag = 0, ACK flag = 1 — handshake complete

What you just did: You watched a TCP 3-way handshake in slow motion, one packet at a time, with every flag visible. This is what happens invisibly in milliseconds on your real network — Packet Tracer shows it at human speed.

Exercise A2 — Simulate a Port Scan manually

A real port scan sends SYN packets to many ports and reads which ones respond with SYN-ACK (open) vs RST (closed). In Packet Tracer, you simulate this with custom PDUs.

Step 1: Go to Realtime Mode. Click the envelope icon at the top right (Add Simple PDU or Add Complex PDU — use Complex).

Step 2: Click PC0 as source, PC1 as destination. In the PDU settings window:

  • Protocol: TCP
  • Source Port: 1234 (random)
  • Destination Port: 80
  • SYN: checked, ACK: unchecked

Step 3: Add another Complex PDU from PC0 to PC1 with Destination Port: 23 (Telnet).

Step 4: Add another with Destination Port: 22 (SSH).

Step 5: Switch to Simulation Mode and step through. Watch each SYN packet travel to PC1. Since PC1 is a basic Packet Tracer PC, it will respond with RST (port closed) because no service is listening. That RST response is exactly what a real port scanner sees and records.

Step 6: Now go to PC1 → Desktop → Services. Enable HTTP. Send the SYN to port 80 again. This time PC1 responds with SYN-ACK — port is open. This is how Nmap works. You just manually did what Nmap automates.

Now open Wireshark on your real machine and do this:

Start capture → open Command Prompt → type:
nmap -sS 192.168.1.1 --top-ports 20

(if nmap isn't installed: nmap.org, download free)

Apply filter: tcp.flags.syn==1 && tcp.flags.ack==0

You'll see the exact same pattern you just built manually in Packet Tracer — rapid SYN packets to different ports. The difference is this is real traffic at real speed. Packet Tracer showed you WHY each packet exists. Wireshark shows you what it looks like at volume.

Lab B — ARP Poisoning Deep Dive

Build this network in Packet Tracer

[PC0 — Attacker]
[PC1 — Victim]        ←→  [Switch]  ←→  [Router — Gateway]
[PC2 — Another user]                      192.168.1.1
  192.168.1.10
  192.168.1.20
  192.168.1.30

Add a Router, connect it to the Switch. Set Router interface IP to 192.168.1.1.

Exercise B1 — Watch normal ARP first

Step 1: Switch to Simulation Mode. Filter to show only ARP packets.

Step 2: Click PC1 → Desktop → Command Prompt → type ping 192.168.1.1

Step 3: Step through the simulation. The VERY FIRST packet generated is NOT the ping — it's an ARP Request. PC1 broadcasts "Who has 192.168.1.1?" to MAC FF:FF:FF:FF:FF:FF (everyone on the subnet).

Step 4: Click that ARP packet envelope. In PDU Details you'll see:

ARP REQUEST
Sender IP: 192.168.1.20
Sender MAC: [PC1's MAC]
Target IP: 192.168.1.1
Target MAC: 00:00:00:00:00:00  (unknown — that's WHY it's asking)

Step 5: Step forward. The Router sends an ARP Reply directly back to PC1:

ARP REPLY
Sender IP: 192.168.1.1
Sender MAC: [Router's MAC]  ← the answer
Target IP: 192.168.1.20
Target MAC: [PC1's MAC]

PC1 stores this in its ARP cache. Now it knows the router's MAC. This is normal, legitimate ARP. You just watched it happen packet by packet.

Step 6: On PC1 → Command Prompt → type arp -a. You'll see the router's IP mapped to its MAC. This is the cache that ARP poisoning attacks.

Exercise B2 — Simulate the ARP Poison manually

Packet Tracer doesn't let you send fake ARP Replies automatically (it's not a real attack tool — it's a learning tool). So you simulate it by understanding what the attacker's packet WOULD look like and then observing the effect.

Step 1: Note PC0's (attacker's) MAC address. Click PC0 → Config → FastEthernet0 → note the MAC address shown.

Step 2: Now here's the attack packet the attacker would send — a fake ARP Reply saying:

ARP REPLY (FAKE — sent by PC0 attacker)
Sender IP: 192.168.1.1   ← claims to be the ROUTER
Sender MAC: [PC0's MAC]  ← but it's actually the attacker's MAC
Target IP: 192.168.1.20  ← sent to the victim
Target MAC: [PC1's MAC]

Step 3: Use the Complex PDU tool. Select ARP protocol. Set it to send from PC0 to PC1 as an ARP Reply. Set the Sender IP as 192.168.1.1 and Sender MAC as PC0's MAC. Send it.

Step 4: Now check PC1's ARP cache: arp -a. The router IP 192.168.1.1 now maps to PC0's MAC address — the attacker. Every packet PC1 tries to send to the internet now goes to PC0 first.

Step 5: Switch to Simulation Mode and ping from PC1 to 8.8.8.8. Watch where the packet goes. It goes to PC0 (attacker) instead of the router. MITM complete.

Now open Wireshark on your real machine:

Start capture → apply filter: arp

Go to your Command Prompt:

arp -a

Note your gateway's MAC. Now watch for any ARP packet that claims your gateway's IP but with a different MAC — that would be a real poisoning attempt. On your home/office network you likely won't see one, but you now know exactly what to look for because you built it yourself.

Lab C — ICMP Flood (DoS Simulation)

Same 3-PC network as Lab B

Exercise C1 — Normal ping vs flood

Step 1: Realtime Mode. PC0 → Desktop → Command Prompt:

ping 192.168.1.20

This sends 4 ICMP Echo Requests and receives 4 Echo Replies. Normal traffic.

Step 2: Now simulate a DoS flood. In Packet Tracer Command Prompt:

ping 192.168.1.20 -n 1000 -l 1000

-n 1000 = 1000 packets, -l 1000 = 1000 bytes each.

Step 3: Switch to Simulation Mode while this runs. Watch PC1 receiving hundreds of ICMP packets. Its buffer fills. Legitimate requests from PC2 trying to reach PC1 get delayed or dropped — this is the DoS effect.

Step 4: Click any ICMP packet envelope. PDU Details shows:

ICMP TYPE: 8 (Echo Request)
CODE: 0
DATA: [payload bytes]

The Type 8 flooding is the attack. Type 0 (Echo Reply) comes back — but during a real flood, the victim eventually can't send Type 0 replies fast enough and drops them.

In Wireshark on your real machine:

Start capture → Command Prompt:
ping -n 100 -l 65500 192.168.1.1

Apply filter: icmp

Go to Statistics → I/O Graph. You'll see the spike your own ping flood creates. On a real DoS attack, this graph shows a sustained vertical wall of packets from an external source. You've now seen it from both the attacker's and defender's perspective.

Lab D — FTP Credential Theft (Plain Text)

Build this network in Packet Tracer

[PC0 — Attacker/Sniffer]  ←→  [Hub]  ←→  [PC1 — Client]
                                  ↓
                            [Server0 — FTP Server]
                              192.168.1.100

IMPORTANT: Use a HUB not a Switch. A hub broadcasts all traffic to all ports — this is how older networks worked and why sniffing was easy. A switch is smarter and only sends traffic to the correct port. But on a hub, PC0 (attacker) receives ALL traffic including the FTP session between PC1 and the server.

Step 1: Set up the FTP server. Click Server0 → Services → FTP. Enable FTP. Note the default username/password (usually cisco/cisco).

Step 2: Switch to Simulation Mode. Filter to show FTP packets only.

Step 3: Click PC1 → Desktop → Command Prompt:

ftp 192.168.1.100

Enter username: cisco and password: cisco

Step 4: Step through packets in Simulation. Click each FTP packet envelope. In PDU Details you will see the application layer data:

FTP: USER cisco
FTP: PASS cisco
FTP: 230 Login successful

The username and password are completely visible in plain text inside the packet. PC0 (the attacker on the same hub) receives these packets too — it can read them just as easily because it's on the same hub.

This is the single most important demonstration for your Power Grid job. If any of your industrial devices still use FTP — and many do — any attacker who gains access to the same network segment can capture credentials this easily, with zero effort.

In Wireshark on your real machine:

Start capture → in Command Prompt:
ftp test.rebex.net
(this is a public FTP test server — safe to use)
Username: demo
Password: password

Apply filter: ftp

You'll see the exact same thing you simulated — username and password in plain readable text in the packet list.

Lab E — DNS Analysis and Anomaly Detection

Same basic network, add a DNS Server

Step 1: Add a Server, enable DNS service. Add an A Record: powergrid.local → 192.168.1.50

Step 2: Set PC1's DNS server to point to your DNS server IP.

Step 3: Simulation Mode, filter DNS only.

Step 4: PC1 → Command Prompt → ping powergrid.local

Step 5: Step through. FIRST packet is a DNS Query:

DNS QUERY
Type: A
Name: powergrid.local

Step 6: Server responds with DNS Reply:

DNS REPLY
Name: powergrid.local
Type: A
Address: 192.168.1.50

Step 7: ONLY NOW does the ICMP ping packet appear — because the IP was unknown until DNS resolved it.

Now simulate DNS tunnelling conceptually: Change your A record name to something like c3RvbGVuZGF0YQ.evil.local. In a real DNS tunnelling attack, malware queries absurdly long subdomains like this to encode stolen data. Packet Tracer shows you the DNS packet structure — the query name field is where the data hides.

In Wireshark on your real machine:

Start capture → PowerShell:
Resolve-DnsName google.com -Type A
Resolve-DnsName google.com -Type MX
Resolve-DnsName google.com -Type TXT

Apply filter: dns

Compare the frame sizes. Type A queries are small. TXT records are larger. Abnormally large DNS frames are your tunnelling indicator.

The Combined Workflow — How to Use Both Tools Together Every Time

Here is the exact process you should follow for every attack you want to deeply understand:

Step 1 — Build it in Packet Tracer. Set up the simplest possible topology that demonstrates the attack. Use Simulation Mode. Step through every single packet. Read the PDU Details of each one. Understand WHY each packet exists.

Step 2 — Draw the packet flow. On paper or a notes app, write down the sequence: what packet comes first, what protocol, what flags, what payload. For example for ARP poisoning: ARP Request → ARP Reply (fake) → victim cache poisoned → traffic redirected.

Step 3 — Capture it in Wireshark. Now do the same thing on your real machine or simulate the equivalent traffic. Apply the corresponding filter. Confirm that what you see in Wireshark matches what you drew from Packet Tracer.

Step 4 — Write the detection rule. For each attack, write your Wireshark filter as a detection rule. Port scan: tcp.flags.syn==1 && tcp.flags.ack==0 && tcp.flags.ack==0. ARP poison: arp.duplicate-address-detected. FTP creds: ftp. This is the beginning of writing your own detection playbook.
