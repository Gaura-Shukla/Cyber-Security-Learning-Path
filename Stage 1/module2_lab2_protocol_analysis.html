Protocols you will encounter every day in security
TCP
TCP
Connection-oriented. Guarantees delivery via handshake + acknowledgements. Every web page, login, and file transfer uses it. Filter: tcp
DNS
DNS
Translates domain names to IPs. Every internet connection starts with a DNS query. Malware hides in unusual DNS. Filter: dns
HTTP
HTTP/S
Web traffic. HTTP = fully readable. HTTPS = encrypted (TLS). You can see HTTPS but cannot read the body. Filter: http tls
ARP
ARP
Maps IP → MAC on local network. Poisoning ARP = MITM attack. No Layer 3 — stays on your subnet only. Filter: arp
ICMP
ICMP
Ping and traceroute. Used in network recon and DoS floods. No ports — works at Layer 3 only. Filter: icmp
FTP
FTP / Telnet
Legacy protocols — passwords travel in plain text. Still found in industrial/power infrastructure. Critical to detect. Filter: ftp telnet
UDP
UDP
Connectionless — no handshake, no guarantee. Used by DNS, DHCP, video streaming. Attackers use it for amplification DDoS. Filter: udp
TCP 3-way handshake — what it looks like in Wireshark
Your machine
192.168.1.x
Server
104.x.x.x :80
Packet 1
SYN →
"I want to connect. Here is my sequence number." Flags: SYN=1, ACK=0. Wireshark filter: tcp.flags.syn==1 && tcp.flags.ack==0
Packet 2
← SYN-ACK
"Got it. I'm ready. Here is MY sequence number." Flags: SYN=1, ACK=1. Wireshark filter: tcp.flags.syn==1 && tcp.flags.ack==1
Packet 3
ACK →
"Acknowledged. Connection established." Data transfer starts from next packet. Flags: SYN=0, ACK=1.
Security relevance at your Power Grid job
A SYN flood attack sends thousands of SYN packets without ever completing the handshake. The server allocates memory for each half-open connection and eventually runs out. Filter tcp.flags.syn==1 && tcp.flags.ack==0 and look for one source IP sending to many destination ports or many times per second — that is your indicator.
Part A — TCP handshake capture
Flush DNS cache, then start a fresh Wireshark capture
PowerShell: ipconfig /flushdns — then immediately start capturing. This ensures the next browser visit generates both DNS AND TCP handshake packets you can observe.
Visit http://neverssl.com — then stop capture after 10 seconds
You need a fresh visit to a never-before-visited site so the full handshake occurs. Apply filter: tcp.flags.syn==1 — you should see 3 packets: SYN, SYN-ACK, ACK in sequence.
Find the SYN packet. Expand TCP layer. Read the Sequence Number.
Every TCP connection starts with a random initial sequence number. The server uses this to order all future packets. Note the value — it is unique to this connection.
Find the SYN-ACK. Confirm Acknowledgement Number = your SYN seq + 1
The server's ACK number equals your sequence number + 1. This proves it received your SYN. This is how TCP guarantees delivery — every byte must be acknowledged.
Part B — DNS dissection
Apply filter: dns — find the query for neverssl.com
You will see two packets: a DNS Query (your machine asking) and a DNS Response (the DNS server answering). The query travels over UDP port 53. Note the transaction ID — it matches in both query and response.
Expand the DNS Response. Read the Answer section — note the IP returned.
Expand Domain Name System → Answers → A record. The IP here is exactly what your browser used to connect. Cross-check: it must match the Destination IP you saw in the TCP handshake SYN packet.
Apply filter: dns and look at the Query Type field
Type A = IPv4 address lookup. Type AAAA = IPv6 lookup. Type MX = mail server lookup. Type TXT = text records. In security, attackers abuse TXT records for data exfiltration — that is DNS tunnelling.
Part C — ARP on your local network
Start a new capture. Apply display filter: arp immediately
Within seconds you will see ARP packets broadcasting "Who has X.X.X.X? Tell Y.Y.Y.Y". This is your machine or others on the subnet discovering MAC addresses. ARP only works within your local network — it never crosses a router.
Find an ARP Reply. Expand it. Note the Sender MAC and Sender IP together.
This is the critical data in ARP. In an ARP poisoning attack, a malicious machine sends fake ARP Replies claiming its MAC address is the router's IP. Every device that accepts this reply routes all traffic through the attacker. One MAC claiming to be many IPs = red flag.

Mirror every lab task in PowerShell
Watch TCP connections form in real time
netstat -an 2
Refreshes every 2 seconds. Start this BEFORE visiting neverssl.com. Watch a new ESTABLISHED connection appear on port 80 — this is the same TCP connection whose handshake you see in Wireshark.
Manually perform a DNS lookup — Type A record
Resolve-DnsName neverssl.com -Type A
This generates one DNS Query + one DNS Response — exactly the two packets you see in Wireshark with the dns filter. The IP returned must match Wireshark's DNS answer section.
Query different DNS record types
Resolve-DnsName google.com -Type MX
Resolve-DnsName google.com -Type TXT
Resolve-DnsName google.com -Type AAAA
Each of these generates different DNS query types. Watch them appear in Wireshark simultaneously. MX = mail servers. TXT = SPF/DMARC email auth records. AAAA = IPv6 addresses.
View your ARP table (devices your machine knows about)
arp -a
Lists every IP-to-MAC mapping cached on your machine. These are the devices you have recently communicated with on your local network. Compare this with the ARP packets you captured — they should match. In an ARP poisoning attack, the attacker's MAC would appear here next to your gateway IP.
Send a ping and watch ICMP in Wireshark simultaneously
ping -n 5 neverssl.com
Run this while Wireshark captures. Apply filter icmp. You will see exactly 5 Echo Request packets (Type 8) and 5 Echo Reply packets (Type 0). The round-trip time in Wireshark's Time column will match the ms values ping reports in the terminal.
Trace the route and watch it in Wireshark
tracert neverssl.com
Each hop generates 3 ICMP packets with a different TTL. In Wireshark with filter icmp, watch packets with TTL=1, TTL=2, TTL=3 appear in sequence — each one reaching one router further. This is how tracert maps the network path to a destination.
5 questions — answer all before moving to Module 3.
1. In a TCP 3-way handshake, what does the server's SYN-ACK packet prove?
The server has sent data successfully to the client
The server received the client's SYN and is ready — it acknowledges the client's sequence number + 1
The connection is now fully established and data transfer begins
2. You work at Power Grid Corp. You see hundreds of SYN packets from one IP to many different destination ports in 2 seconds. What is happening?
Normal web browsing — browsers open many connections simultaneously
A DNS amplification attack targeting the DNS server
A port scan — an attacker is probing which ports are open on your network
3. DNS uses which transport protocol and port?
TCP port 80
UDP port 53 (TCP port 53 for large responses)
ICMP — no port number
4. In Wireshark, you see the same IP address appearing in multiple ARP Replies claiming to be different IPs including your gateway. What is this?
Normal ARP behaviour — one device can have multiple IP addresses
ARP poisoning — an attacker is attempting a man-in-the-middle attack by redirecting traffic through their machine
A DHCP server assigning new addresses to devices
5. ARP operates at which OSI layer, and why can it NOT cross a router?
Layer 3 — routers strip IP headers before forwarding
Layer 2 — ARP works with MAC addresses which are only valid within a local subnet; routers operate at Layer 3 and do not forward Layer 2 broadcasts
Layer 4 — it requires a TCP connection which routers block by default
